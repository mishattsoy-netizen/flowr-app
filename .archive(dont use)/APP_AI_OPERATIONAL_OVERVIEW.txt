FLOWR AI BOT — OPERATIONAL OVERVIEW (v1.0)
========================================

1. SYSTEM ARCHITECTURE & PHILOSOPHY
-----------------------------------
Flowr AI is built on a modular "Pipeline" architecture. Instead of a single monolithic model call, every user request is processed through a sequence of specialized AI "Chains." This allows the system to use the right model for the right task, maintain high quality through reasoning steps, and optimize token usage.

The system is governed by:
- The Brain (bot_brain_entries): Factual knowledge, tone refinements, and rules.
- Bot Settings (bot_settings): Core personality, answer styles, and restrictions.
- Pipeline Prompts (pipeline-*.txt): Structural roles for each stage of the request.

2. THE REQUEST LIFECYCLE (PHASES)
---------------------------------

PHASE 0: ADVISOR (Pre-flight)
- Role: Decides if the user request is clear enough to answer.
- Logic: If the request is ambiguous or missing critical context (e.g., "write an essay" without a topic), the Advisor stops the pipeline and asks EXACTLY ONE clarifying question.
- PASS Condition: If the request is clear, it outputs "PASS" followed by [CLASSIFIER_NOTES].
- Classifier Notes: These notes synthesize the user's intent and any previous clarifications into a focused instruction for the next phase, preventing "context forgetfulness."

PHASE 1: INTENT CLASSIFICATION
- Role: Routes the request to the correct execution path.
- Priority:
    1. Intent Tags: Explicit tags like /search or /image are trusted immediately.
    2. Keywords: Fast-path matching based on database-defined regex patterns.
    3. AI Classifier: A model call (usually a fast model like Gemini Flash) that selects from:
       - FAST_SIMPLE: Greetings, lookups, short answers.
       - MEDIUM_THINKING: Explanations, summaries, standard tasks.
       - COMPLEX_THINKING: Deep reasoning, analysis, multi-step logic.
       - CODING: Code generation and technical tasks.
       - WEB_SEARCH: Real-time information retrieval.
       - DEEP_RESEARCH: Multi-source synthesis and comprehensive research.
       - IMAGE_GEN: Visual content creation.
       - VISION: Image analysis (triggered automatically if an image is detected).
       - MULTI_CHAIN: Complex requests requiring orchestration.

PHASE 2: ORCHESTRATION (Only for MULTI_CHAIN)
- Role: Plans a sequence of steps to solve a complex request.
- Logic: The Orchestrator plans the minimal needed chains (e.g., [WEB_SEARCH, CODING, MEDIUM_THINKING]) and sets a specific goal for each step.
- System Automaticity: The system automatically adds a THINKING step and an OUTPUT step to every planned sequence.

PHASE 3: PIPELINE EXECUTION
- Role: Runs the planned chains sequentially.
- Context Accumulation: Each step's output is truncated and passed to the next step as [PIPELINE CONTEXT], ensuring each specialized model knows what the previous models found.

PHASE 4: THINKING (Reasoning Layer)
- Role: Reviews all collected data, catches errors, and chooses the final answer direction.
- Self-Correction: If the Thinking step detects a major gap, it can request a "Correction" (re-running a search or research step) before proceeding.

PHASE 5: FINAL OUTPUT (Synthesis)
- Role: Generates the final user-facing response.
- Context Injection: This step receives:
    - [SESSION MEMORY SUMMARY] (if the conversation is long).
    - [GLOBAL PROMPT] (Brain entries + Core rules).
    - [PIPELINE CONTEXT] (Data from all previous steps).
    - [THINK CHAIN DIRECTION] (Specific instructions on how to write the answer).
    - [ATTENTION BLOCK] (Specific context if replying to a specific message).

3. ROUTING & STABILITY
----------------------
- Chain Router: Manages a list of models for each intent category. If the primary model (e.g., Groq) fails, it automatically falls back to the next one (e.g., Gemini).
- Circuit Breaker: Models that fail are marked and skipped for 1 minute to prevent hanging the UI.
- Provider Support: Google (Gemini), Groq, OpenRouter, Cloudflare AI, HuggingFace, Pollinations, and Local (Ollama).

4. CONTEXT & MEMORY MANAGEMENT
------------------------------
- Conversation History: The last 20 turns (User/Model pairs) are always provided to every step.
- Date/Time Injection: Every final answer chain is injected with the current date and time to handle temporal queries.
- Session Compaction (Summarization): When a session reaches 80% of its context limit (e.g., 32k tokens), a background process summarizes the history into a "Distilled Summary." This summary is then injected into the system prompt, allowing the bot to "remember" long-term goals while staying within token limits.

5. MODES & SCENARIOS
--------------------
- Default Mode: Standard personality and behavior.
- Pro Mode: Can have different prompts, models, and restrictions (configured in Admin).

SCENARIOS:
- User says "Hi": Advisor PASS -> Classifier FAST_SIMPLE -> Direct Answer.
- User says "Search for latest AI news": Advisor PASS -> Classifier WEB_SEARCH -> Search Output -> Synthesis.
- User says "Write a React component for a chart using data from Yahoo Finance": 
  Advisor PASS -> Classifier MULTI_CHAIN -> Orchestrator plans [WEB_SEARCH, CODING] -> Pipeline runs -> Thinking step -> Final synthesis.
- User attaches image: System bypasses standard routing -> Vision Chain -> Analysis.

6. PROMPT STRUCTURE (THE "BRANCH")
----------------------------------
The final system prompt is a "branch" of several blocks:
1. [CURRENT CONTEXT] (Date/Time)
2. [SESSION MEMORY SUMMARY] (If available)
3. [GLOBAL COMPILED PROMPT] (Core Rules + Brain Facts/Tone)
4. [PIPELINE CONTEXT] (If steps were run)
5. [THINK CHAIN DIRECTION] (If thinking was enabled)
6. [INTERNAL ROLE PROMPT] (e.g., "You are the final synthesis step...")
7. [RESTRICTIONS] (Hard limits defined in settings)

7. SPECIALIZED SUBSYSTEMS
-------------------------

ROADMAP ARCHITECT
- Role: A senior-level project planning assistant embedded in the dashboard.
- Logic: Uses its own dedicated classification and routing (roadmap_router_chains).
- Capabilities: Decomposes features into phases, tasks, and production-ready agent_prompts.
- Output: Generates structured [ROADMAP_ACTION] blocks that the UI parses into interactive elements.
- Clean-up: Features a robust response cleaner that strips internal "thinking" tags (<thought>, <think>) and extracts clean user-facing answers.

8. EXTERNAL INTERFACES & UTILITIES
----------------------------------
- Telegram Integration: The bot can be accessed via Telegram, using the same routing and pipeline logic (src/lib/bot/telegram.ts).
- Notifications: The system can trigger push notifications or internal alerts based on AI events (src/lib/bot/notifications.ts).
- Usage Guard: Monitors usage patterns and enforces safety/rate limits (src/lib/bot/usageGuard.ts).

9. ANALYTICS & TRACKING
-----------------------
- Model Usage tracking: Every successful model call is tracked in the database (increment_model_usage) for cost analysis and performance monitoring.
- Routing Trace: The system maintains a detailed trace of which models were attempted, which failed, and which eventually succeeded, visible in the Admin Message Logs.
