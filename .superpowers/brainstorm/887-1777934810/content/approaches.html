Implementation Approaches

3 ways to architect the mode system + context controls

A

Mode as a prompt overlay ★ Recommended

Each mode stores its own 5 settings blocks in the existing bot_settings table with a new mode column (default / think / pro). The compiled prompt system gets a mode parameter — on send, the active mode's compiled prompt is injected instead of the global one. Admin gets 3 pages mirroring the current settings page. Context controls added as a new settings section. Compaction chain stored as structured JSON in a new bot_compaction_config table.

Pros

  • Minimal schema change — reuses existing tables
  • Each mode compiles independently — fast lookup
  • Clean separation, easy to extend

Cons

  • 3 compiled prompts to keep in sync
B

Mode as a full profile entity

Create a new bot_modes table that owns everything — settings blocks, classifier prompt, compiled output — as a self-contained profile. Switching modes swaps the entire profile. Admin gets 3 separate pages each managing a mode profile. More normalized but requires a new abstraction layer.

Pros

  • Cleaner data model long-term
  • Easy to add more modes later

Cons

  • Larger schema change
  • More migration work
  • Breaks existing settings page — needs refactor
C

Mode as a runtime injection

Keep a single settings table, but store mode-specific prompt overrides as separate fields applied at runtime on top of the base settings. The chat route receives the mode and dynamically assembles the prompt by layering overrides. Lightest DB change but more complex runtime logic.

Pros

  • Smallest DB footprint

Cons

  • Complex prompt assembly at runtime
  • Harder to preview/debug per-mode prompts
  • No clean per-mode compilation
